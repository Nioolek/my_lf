# 为什么 LangGraph 要分成 Python + Go 两个服务？

## 核心问题

官方 Postgres Runtime 选择了 **Python + Go 双进程 + gRPC 通信** 的架构，而不是像我们自研的 `postgres_py` 那样用一个纯 Python 进程搞定一切。这明显增加了复杂性：

- 多了一个进程要部署和维护
- 引入了 gRPC 通信层，多了序列化/反序列化开销
- 两个进程共享数据库，需要协调数据一致性
- 调试时需要跨语言追踪问题

**那为什么官方还要这么做？**

---

## 1. 先看清楚：两种架构到底是什么

### 单进程架构（postgres_py）

```
┌─────────────────────────────────────┐
│           Python 进程               │
│                                     │
│  FastAPI ──→ ops.py ──→ PostgreSQL  │
│  Worker  ──→ graph.astream()        │
│           ──→ Checkpointer ──→ PG   │
│           ──→ Redis Pub/Sub         │
│                                     │
└─────────────────────────────────────┘
```

**一条线到底，简单直接。**

### 双进程架构（官方）

```
┌──────────────────────┐       gRPC        ┌──────────────────────┐
│    Python 进程        │ ←──────────────→  │    Go 进程           │
│                      │                    │                      │
│  FastAPI             │                    │  gRPC Server         │
│  Worker (图执行)      │                    │  元数据 CRUD ──→ PG  │
│  Checkpointer ──→ PG │                    │  事件分发 ──→ SSE    │
│                      │                    │  TTL 清理            │
└──────────────────────┘                    └──────────────────────┘
```

**两个进程各管一摊，通过 gRPC 沟通。**

---

## 2. 单进程架构的好处

### 2.1 简单

这是最大的优势。没有 gRPC，没有跨进程通信，没有 protobuf 序列化。

```python
# 单进程：直接函数调用
thread = await Threads.get(conn, thread_id)

# 双进程：需要序列化 → 网络传输 → 反序列化
request = pb.GetThreadRequest(thread_id=pb.UUID(value=str(thread_id)))
response = await client.threads.Get(request)
thread = proto_to_thread(response.thread)
```

单进程是 **1 次函数调用**，双进程是 **3 次操作**（序列化 + gRPC + 反序列化）。

### 2.2 部署简单

```bash
# 单进程
pip install langgraph-runtime-postgres-py
python -m langgraph_runtime

# 双进程
docker compose up  # 需要两个容器 + 网络 + 健康检查
```

单进程不需要 Docker，不需要配置 gRPC 地址，不需要担心 Go 进程挂了。

### 2.3 调试简单

单进程中，所有代码都是 Python。你可以：

- 在 IDE 中设断点，从 HTTP 请求一路追踪到 SQL 查询
- 用 `print()` 看到完整的调用链
- 用 Python profiler 找到性能瓶颈

双进程中，如果 gRPC 调用出问题，你得：

- 看 Python 端的日志
- 看 Go 端的日志
- 对照两边的时间戳
- 而且 Go 端是闭源二进制，你连代码都看不到

### 2.4 延迟低

```
单进程：Python → 函数调用 → SQL → PostgreSQL
双进程：Python → gRPC序列化 → 网络 → Go → SQL → PostgreSQL
         + 网络延迟（localhost ~0.1-0.5ms）
         + protobuf 序列化/反序列化 (~0.01-0.1ms)
```

单次请求差 0.1-1ms，但在高并发下会累积。

### 2.5 代码量少

自研 `postgres_py` 的 `ops.py` 约 800 行，替代了官方的 gRPC ops 层 + Go 服务。整个项目少了一个进程的代码。

---

## 3. 双进程架构的好处

### 3.1 性能隔离 —— 图执行不影响 API 响应

这是**最核心的原因**。

图执行（`graph.astream()`）是 CPU 密集 + I/O 密集的操作。一个 LLM 调用可能需要几十秒，期间占用事件循环。如果图执行和 API 服务在同一个进程：

```python
# 用户正在执行一个复杂的 Agent 图（耗时 30 秒）
result = await graph.astream(input, config)  # 这会占用 Worker 的事件循环

# 此时另一个用户想创建一个 Thread
# → FastAPI 请求排队等待
# → API 响应延迟飙升
```

双进程架构下：

```
Python 进程 1（API）：专门处理 HTTP 请求，创建 Thread/Run 只需要一次 gRPC 调用（~1ms）
Python 进程 2（Worker）：执行图，跑 30 秒也不影响 API 进程
Go 进程：专门处理元数据 CRUD，响应速度稳定
```

**关键：API 响应和图执行完全隔离，互不影响。**

### 3.2 独立扩展

```
场景：100 个并发图执行，但只有少量 API 请求

单进程：需要启动 100 个 Python 进程（每个都能处理 API + 执行图）
双进程：启动 100 个 Worker + 1 个 API + 1 个 Go → 节省资源

场景：大量 API 请求，但只有少量图执行

单进程：需要启动更多 Python 进程（即使大部分只处理 API）
双进程：启动多个 API 实例 + 少量 Worker → 按需分配
```

双进程架构允许你**按角色独立扩展**：

| 组件 | 扩展场景 |
|------|---------|
| API 实例 | API 请求量大时加实例 |
| Worker 实例 | 图执行并发高时加实例 |
| Go 实例 | 数据库操作是瓶颈时加实例 |

单进程只能整体扩展，浪费资源。

### 3.3 故障隔离

```
场景：用户的图代码有 bug，导致 Worker OOM 崩溃

单进程：整个进程挂了 → API 也无法响应 → 所有用户受影响
双进程：Worker 崩溃 → Go 服务和 API 正常 → 其他用户不受影响 → 自动重启 Worker
```

图执行是最容易出问题的环节（用户代码不可控），把它隔离到独立进程是合理的。

### 3.4 Go 的性能优势

Go 在以下方面确实比 Python 快：

| 操作 | Python (asyncpg/psycopg3) | Go (pgx) | 差异 |
|------|--------------------------|----------|------|
| 简单 SQL 查询 | ~2ms | ~0.5ms | 4x |
| 高并发连接 | 受 GIL 限制 | 无限制 | 显著 |
| 内存占用 | ~100MB/进程 | ~20MB/进程 | 5x |
| 连接池管理 | 较重 | 轻量 | 显著 |

对于元数据 CRUD 这种简单但高频的操作，Go 的优势明显。

### 3.5 商业保护

LangChain 将核心持久化逻辑放在 Go 中闭源：

```
开源部分：
  - Python API 层（你知道它怎么调用 Go）
  - Python Worker（你知道图怎么执行）

闭源部分：
  - Go 服务（你不知道它怎么存数据、怎么管理队列）
  - 数据库 Schema 优化（Go 端可能有自己的迁移逻辑）
  - 企业级特性（集群、监控等）
```

这意味着用户**必须使用官方 Docker 镜像**，无法自行构建完整的 Postgres Runtime。

### 3.6 多语言支持

官方还有 JavaScript 版本的 Worker（用于执行 JS/TS 编写的图）。双进程架构下：

```
Python Worker：执行 Python 图
JS Worker：执行 JavaScript 图
Go 服务：统一处理元数据（与 Worker 语言无关）
```

如果用单进程架构，就得为每种语言实现一套完整的元数据 CRUD 逻辑。

## 4. 为什么这样分层？

官方的分层逻辑是按**职责**划分，不是按**技术栈**划分：

```
                    ┌─────────────────────────────┐
                    │        接口层 (API)          │  ← Python FastAPI
                    │   接收 HTTP 请求，路由分发     │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────┴──────────────┐
                    │                              │
           ┌───────▼───────┐              ┌───────▼───────┐
           │   计算层 (Worker)  │              │   数据层 (Go)    │
           │   执行用户代码      │              │   管理持久化数据   │
           │   图的状态机       │              │   任务调度        │
           │   LLM 调用       │              │   事件分发        │
           └───────┬───────┘              └───────┬───────┘
                   │                              │
                   └──────────────┬───────────────┘
                                  │
                    ┌─────────────▼──────────────┐
                    │      存储层 (PostgreSQL)     │
                    │   共享数据库                  │
                    └────────────────────────────┘
```

**分层原则：**

1. **计算层和数据层必须分离** —— 因为图执行（计算）和元数据管理（数据）的资源需求完全不同
2. **数据层用 Go** —— 因为 CRUD 操作是 I/O 密集型的，Go 的并发模型更合适
3. **计算层必须用 Python** —— 因为用户代码是 Python，LangChain 是 Python
4. **共享存储** —— 两个层需要访问相同的数据，通过共享 PostgreSQL 实现

## 5. 什么时候用单进程？什么时候用双进程？

### 选择单进程（postgres_py）的场景

- **团队规模小**，不需要独立扩展各组件
- **部署环境简单**，不想引入 Docker
- **开发和调试为主**，纯 Python 更方便
- **需要完全掌控代码**，不接受闭源依赖
- **负载不高**，单进程足够应对
- **学习研究**，需要理解每个细节

### 选择双进程（官方）的场景

- **生产环境**，需要高可用
- **高并发**，需要独立扩展 Worker 和 API
- **多语言图**，需要同时支持 Python 和 JS Worker
- **企业级需求**，需要官方支持
- **团队有运维能力**，能管理多进程部署
- **愿意使用闭源组件**，不介意 Go 服务不透明

## 6. 本质：这是一次工程权衡

| 维度 | 单进程胜 | 双进程胜 |
|------|---------|---------|
| 简单性 | ✅ | |
| 部署难度 | ✅ | |
| 调试体验 | ✅ | |
| 延迟（单次请求） | ✅ | |
| 代码量 | ✅ | |
| 性能隔离 | | ✅ |
| 故障隔离 | | ✅ |
| 独立扩展 | | ✅ |
| 高并发吞吐 | | ✅ |
| 多语言支持 | | ✅ |
| 商业保护 | | ✅ |

**一句话总结：**

> 单进程架构追求**简单**，双进程架构追求**健壮**。对于 LangChain 来说，作为平台服务商，健壮性比简单性更重要——所以他们选择了双进程。但对于大多数自托管用户来说，简单性可能更有价值——这就是 `postgres_py` 存在的意义。

## 7. 一个类比

把 LangGraph Runtime 想象成一家餐厅：

**单进程（postgres_py）**：
- 一个厨师包揽一切：接单、做菜、收银、打扫
- 简单，一个人就能开
- 客人多时忙不过来
- 厨师病了，餐厅关门

**双进程（官方）**：
- 前台（API）负责接单
- 厨房（Worker）负责做菜
- 经理（Go）负责调度、记账、采购
- 各司其职，可以独立扩展
- 厨师病了，前台还能接单，经理还能调度
- 但需要协调沟通（gRPC），增加管理成本

小餐馆用第一种模式就够，大型连锁必须用第二种。
